# Contributing to ParishPro

Thank you for your interest in contributing to ParishPro! This document
explains how to get involved, find work, and submit contributions.

## Who Can Contribute

Anyone is welcome to contribute regardless of experience level. We have
work available across planning, design, frontend, backend, and
documentation. If you are new to open source this is a great place to
start.

## Before You Start

1. Read the [meta repo README](README.md) to understand the project
   architecture and current status
2. Join our [Discord](https://discord.gg/DubATk2PKq) to introduce
   yourself and ask questions
3. Browse the [project board](https://github.com/orgs/parishpro-app/projects/1)
   to find available work

## Finding Work

All tasks are tracked as GitHub Issues on the project board. Use labels
to find work that matches your skills:

- `good first issue` - ideal for new contributors
- `help wanted` - actively seeking contributors
- `frontend` - work in the parishpro-web repo
- `backend` - work in the parishpro-api repo
- `documentation` - writing and planning work in the meta repo
- `design` - UI/UX work

## Claiming an Issue

Before starting work on an issue, comment on it to let others know you
are picking it up. This prevents duplicate work. A maintainer will
assign it to you.

Do not open a pull request for an issue that is already assigned to
someone else.

## Workflow

1. Fork the relevant repository
2. Clone your fork locally
3. Create a branch using this naming convention:
   `issue-{number}-short-description`
   Example: `issue-7-api-specification`
4. Do the work
5. Commit your changes following the commit message standards below
6. Push to your fork
7. Open a pull request against the `main` branch of the upstream repo
8. Reference the issue number in your pull request description using
   `Closes #7` so the issue closes automatically when merged

## Commit Message Standards

Use clear, descriptive commit messages in the imperative tense:

- `Add scheduling algorithm specification`
- `Fix availability constraint logic`
- `Update API endpoint for sub requests`

Avoid vague messages like `fix stuff`, `updates`, or `wip`.

## Pull Request Standards

- One pull request per issue
- Keep pull requests focused - do not bundle unrelated changes
- Describe what you changed and why in the PR description
- Link the issue using `Closes #number`
- A maintainer will review within a reasonable timeframe

## Code Style

Code style standards will be documented here once the tech stack is
finalized and development begins.

## Documentation Contributions

Planning documents are drafted in Google Drive and converted to markdown
once finalized. If you want to contribute to a planning document, comment
on the relevant issue and the project lead will share the working draft
with you.

## Questions

Ask questions on [Discord](https://discord.gg/DubATk2PKq) in the
appropriate channel. Please do not open issues just to ask questions.

## Code of Conduct

All contributors are expected to follow our
[Code of Conduct](CODE_OF_CONDUCT.md). Be respectful, inclusive, and
constructive in all interactions.
